这份文档是一份**微商系统课程设计指南**，核心是教新手从零开始搭建一个简单的微商软件，并明确了技术要求、实现步骤和考核标准。下面会用“大白话+拆解细节”的方式，把每个部分讲清楚，确保新手能理解。


# 一、背景描述：先搞懂“微商系统”是什么
这部分先帮你建立对“微商系统”的基础认知，避免一开始就陷入技术细节。

| 核心点 | 新手理解 | 例子/补充 |
|--------|----------|-----------|
| 类似的工具 | 我们要做的系统，和手机上的“购物APP”功能很像 | 比如拼多多、微店、有赞（这些都是你可能用过的微商/电商工具），甚至和手机淘宝、京东的核心逻辑一致——都是“看商品、买东西” |
| 本质 | 不是单纯的“卖东西”，而是“靠社交拉人，再做零售” | 比如你在朋友圈发商品链接（社交引流），朋友点进去买（零售交易），这就是“社交+零售”的核心 |
| 常见模式 | 做微商的三个关键步骤，也是系统要支持的核心流程 | 1. 引流：让更多人看到你的商品（比如系统显示热门商品）；2. 吸粉：留住潜在客户（比如系统记录用户收藏）；3. 交易：完成购买（比如购物车、结算功能） |


# 二、C/S框架：一种“客户端+服务器”的软件结构
“框架”就是软件的“骨架”——先确定骨架，再填功能。C/S框架是两种核心骨架之一，全称“Client/Server（客户端/服务器）”。

## 1. 核心特点：“两端都要装程序”
- 客户端：你用的“软件本体”（比如电脑上的QQ、微信），必须在自己的设备（电脑/手机）上安装才能用。
- 服务器：后台的“大电脑”（比如微信的后台服务器），也需要安装专门的程序，负责存储数据（比如你的聊天记录）、处理请求（比如你发消息给朋友，服务器帮你转发）。

## 2. 分层结构（MVC模式简化版）
新手不用纠结“MVC”的专业定义，只要理解“软件分了几层，每层干不同的活，互不干扰”——就像工厂里“生产部（做产品）、销售部（卖产品）、仓库（存产品）”分工明确。

C/S框架的分层和通信流程如下（从“你操作”到“系统响应”的全路径）：
```
客户端（你用的电脑）          服务器（后台大电脑）          数据库（存数据的“仓库”）
┌───────────┐               ┌───────────┐               ┌───────────┐
│    UI层   │  你看到的界面  │    UI层   │  服务器管理界面 │           │
│  （按钮、  │ ←→ 操作指令   │  （管理员  │ ←→ 管理指令   │           │
│   商品图） │               │  看数据）  │               │           │
├───────────┤               ├───────────┤               ├───────────┤
│  业务层   │  处理你的操作  │  业务层   │  处理核心逻辑  │  持久层   │
│  （点“加  │  （比如判断你  │  （比如算  │  （比如验证你  │  （把“你  │
│  购物车”） │  有没有登录）  │  折扣）    │  的订单金额）  │  买了商品” │
└───────────┘               └───────────┘               │  存进数据库） │
        ↓                         ↓                     └───────────┘
        └───────── TCP/IP通信 ───────┘                    ↑
                         （比如你发的请求、服务器的回复）    │
                                                          │
                                                  （从数据库读数据给服务器）
```

简单说：你在客户端点“买东西”，指令通过TCP/IP（一种网络传输规则）传给服务器，服务器算完价格、查完库存，再把结果发回给你，同时把交易记录存进数据库。


# 三、B/S框架：另一种“浏览器+服务器”的结构
和C/S对应，B/S框架是“Browser/Server（浏览器/服务器）”，更常见于网页版软件。

## 1. 核心特点：“只装服务器，客户端用浏览器”
- 客户端：不用装专门软件，直接用Chrome、Edge等浏览器访问（比如你用浏览器逛淘宝网页版，不用装淘宝APP）。
- 服务器：需要装程序，还多了个“Web容器”（比如Tomcat，相当于“网页的管家”，负责把服务器的内容变成浏览器能看懂的网页）。

## 2. 分层结构（和C/S类似，但有2个关键区别）
```
浏览器（你用的Chrome）       服务器（后台大电脑）          数据库（存数据）
┌───────────┐               ┌───────────┐               ┌───────────┐
│    UI层   │  网页界面      │    UI层   │  服务器管理界面 │           │
│  （网页上  │ ←→ 网页请求   │  （同C/S） │               │           │
│   的按钮） │               │          │               │           │
├───────────┤               ├───────────┤               ├───────────┤
│  （无单独  │  浏览器只负责  │  业务层   │  处理核心逻辑  │  持久层   │
│   业务层） │  “显示网页”    │  （同C/S） │               │  （同C/S） │
└───────────┘               └───────────┘               └───────────┘
        ↓                         ↓                     ↑
        └───────── HTTP通信 ───────┘                    │
                 （网页的传输规则，比如你输网址访问）      │
                                                          │
                                                  （读数据给服务器）
```

## 3. C/S和B/S的新手区分
| 对比项 | C/S框架 | B/S框架 | 我们课程用哪个？ |
|--------|---------|---------|------------------|
| 客户端 | 要装专门程序（比如QQ） | 不用装，用浏览器 | 文档后续重点讲C/S，但也提了B/S的思路 |
| 通信方式 | TCP/IP（更稳定，适合复杂操作） | HTTP（网页常用，简单） | 课程用TCP/IP（因为要练“协议设计”） |
| 适用场景 | 功能复杂的软件（比如游戏、专业工具） | 简单的网页应用（比如网页版查成绩） | 微商系统需要“购物车、折扣计算”，C/S更合适 |


# 四、微商系统：核心功能、性能、架构
这部分是课程设计的“核心任务”——明确你要做的系统到底有哪些功能、要达到什么性能、用什么架构实现。

## 1. 核心功能（必须实现的“能用”的功能）
| 功能模块 | 具体要求 | 新手理解 |
|----------|----------|----------|
| 查看产品列表 | 显示所有商品（比如名称、价格、图片） | 就像你打开淘宝，首页显示一堆商品，这是最基础的功能 |
| 购物车 | 支持“添加商品、删除商品、修改数量” | 你看中的商品先放进购物车，最后一起结账，避免买一个付一次钱 |
| 促销策略（重点难点） | 支持多种优惠方式，还要用“反射+策略+装饰”设计模式 | 举个例子：<br>- 打折：统一9折，或第2件9折、第3件8折（甚至免单）<br>- 降价：直接减10元<br>- 券：用优惠券再减5元<br>“设计模式”不用怕，简单说就是“让代码更灵活，改优惠规则时不用全改” |
| 结算 | 计算购物车总金额（要扣掉优惠），完成支付（不用真付钱，模拟流程即可） | 比如你买2件商品，原价100元，用了10元券，结算时付90元，系统记录“你付了钱” |
| 售后服务 | 支持退货（模拟“申请退货、确认退货”流程） | 比如你买的商品不合适，在系统里点“退货”，服务器确认后，把“已退货”记录存进数据库 |
| UI风格选择 | 可以换不同的界面风格（比如浅色/深色主题，按钮位置调整） | 就像手机换壁纸，不影响功能，只是看起来不一样 |

## 2. 性能要求（“能用”还要“好用”）
新手不用追求“高并发”，但要理解“为什么要考虑这些”：
- 并发：比如“双11”大家同时买东西，系统不能卡（课程里不用做这么复杂，只要模拟“2个用户同时结账”不报错即可）。
- 互斥：比如“限量商品”（像爱马仕限量包、Costco的限量零食），不能让2个用户同时买到最后1件（系统要判断“库存只剩1时，只有先下单的人能买到”）。
- 通信方式：同步/异步（简单说：同步=你发请求后，等服务器回复再做下一步；异步=你发请求后，不用等，继续做别的，服务器回复了再通知你）。
- 分层设计：就是之前说的“UI层、业务层、持久层”分开，比如你想改界面按钮，不用动“计算折扣”的代码（这叫“柔性”，灵活不死板）。
- 软件工程生命周期：按“先想需求→再设计架构→最后写代码”的步骤来，不要上来就瞎写（避免写一半发现功能漏了，要全部重改）。

## 3. 系统架构（把之前的C/S框架落地）
这部分明确了“用什么技术实现各层”，新手不用全懂，先知道“要准备哪些工具”：
| 分层/模块 | 技术选择 | 新手理解 |
|-----------|----------|----------|
| 客户端 | 可以用Java或C++（不限制语言，选你熟悉的） | 比如你学过C++，就用C++写客户端界面；学过Java，就用Java写 |
| 服务器 | 推荐用Java（文档里提了“Java支持”） | 服务器要处理“算折扣、存数据”，Java比较稳定，资料也多 |
| 持久层 | Hibernate或MyBatis（两种“数据映射工具”） | 数据库是“非面向对象”的（比如存的是“表格”），而Java/C++是“面向对象”的（比如代码里是“商品类”），持久层的作用就是“把代码里的‘商品’变成数据库里的‘表格行’”，不用你手动写复杂代码 |
| 数据库 | MySQL或SQLite（两种常用数据库） | 就是存数据的“仓库”，比如商品信息（名称、价格）、用户信息（账号、密码）、销售记录（谁买了什么），都存在这里 |
| 通信 | TCP/IP协议（要自己定义“指令编码”） | 比如你在客户端点“加购物车”，要发一个“10020001”的指令给服务器（1002代表“加购物车”，0001代表“1号商品”）；服务器收到后，回复“20020001”（代表“1号商品加购物车成功”） |

## 4. 服务器关键模块详解（新手重点看）
### （1）业务层（异步通信）
用“观察者模式+消息队列”实现，新手不用懂专业术语，看例子：
- 场景：你买完东西（完成交易），系统要自动给你发“优惠短信”、更新“库存”、记录“销售数据”。
- 实现逻辑：
  1. “消息队列”：把“发短信、更库存、记数据”这三个任务排成队，一个一个做（避免同时做导致混乱）。
  2. “观察者模式”：一旦“交易完成”这个事件发生，就自动触发队列里的任务（不用你手动点“发短信”，系统自己做）。

### （2）持久层
核心作用：“解决代码和数据库的‘语言差异’”——
- 数据库：用“表格”存数据（比如“商品表”有“商品ID、名称、价格”三列）。
- 代码：用“类”存数据（比如Java里的`Goods`类，有`id`、`name`、`price`三个属性）。
- 持久层：帮你把`Goods`类的对象，自动变成“商品表”里的一行数据（不用你写复杂的数据库命令）。

### （3）数据库
新手不用学太深，先知道这3点：
1. 本质：就是电脑上的“特殊文件”，专门用来存结构化数据（比你用Excel存数据更稳定、更快）。
2. 结构：数据库→表→列/行（比如“微商数据库”里有“商品表”“用户表”，“商品表”里每一行是一个商品）。
3. 用什么操作：SQL语言（简单的命令，比如`SELECT * FROM 商品表`就是“查所有商品”，课程只需要初步了解）。

### （4）客户端
核心任务：“让用户能操作”，具体做3件事：
1. 显示界面：比如商品列表、购物车、结算页（UI层）。
2. 接收操作：比如用户点“加购物车”“结算”（把操作转成指令发给服务器）。
3. 显示结果：比如服务器回复“结算成功”，客户端就显示“购买完成”。


# 五、涉及新技术：新手需要学什么
文档列了“必须接触的技术”，不用全精通，先入门：
| 技术 | 学习重点 | 新手理解 |
|------|----------|----------|
| 多层架构 | 理解“UI层、业务层、持久层”各自的作用，怎么配合 | 就是“分工”，记清“谁做什么”，别把代码写混 |
| 数据库+持久层 | 会用MySQL/SQLite建简单的表，会用Hibernate/MyBatis做简单的“数据映射” | 先学“怎么建表”，再学“怎么用工具把代码里的数据存进表” |
| UML | 画简单的“用例图”“类图”（比如画“用户买商品”的流程，画“Goods类”的属性） | 就是“画图”，让别人（包括老师）看懂你的系统设计，不用画太复杂 |
| SQL语言 | 会写“查数据（SELECT）、插数据（INSERT）、改数据（UPDATE）、删数据（DELETE）” | 这4个命令是基础，比如`INSERT INTO 商品表 VALUES (1, "苹果", 5)`就是“加一个1号商品‘苹果’，价格5元” |
| 同步/异步通信 | 理解“同步等回复，异步不等”，会用TCP/IP发简单指令 | 比如你发“加购物车”指令后，等服务器回复“成功”再继续（同步）；发“退货”指令后，不用等，继续逛商品（异步） |
| 跨语言（可选） | 比如服务器用Java，客户端用C++（不用必须做，选一种语言也可以） | 就是“不同语言写的程序能互相通信”，课程不强制，新手可以先专注一种语言 |


# 六、延伸思考：不用做，但要了解的概念
这部分是“拓展视野”，不用实现，知道是什么就行：
- Web编程：文档不建议新手做（因为网页封装太狠，你看不到底层逻辑，不利于学架构）。
- APP和HTML5：APP是手机上装的程序（比如微信APP），HTML5是做网页的技术（比如微信里的“小程序网页”用HTML5做）。
- 安卓与iOS：手机的两个系统，课程不用做手机版，只做电脑版。
- 小程序：比如微信小程序，本质是“在APP里运行的轻量级网页”，不用装，点开就能用。
- Web Service/Web容器：Web容器是“跑网页的工具”（比如Tomcat），Web Service是“不同网页/软件之间通信的规则”，课程用不到。


# 七、约定：明确“不用做什么”，避免走弯路
这部分是“减负”，新手重点看“不用做的事”，专注核心任务：
1. 只做PC版：不用开发手机APP，电脑上能运行就行。
2. 不用多店：系统里只有“一个商家”（比如你自己的店），不用支持“多个商家入驻”。
3. 要多用户：支持“多个买家”（比如A用户、B用户同时登录买东西）。
4. 语言不限：选你熟悉的语言（Java、C++、Python都可以）。
5. 可以发挥：除了要求的功能，你还可以加自己的想法（比如“商品收藏”“历史订单”）。
6. 独自完成：不分组，一个人做，避免依赖别人。


# 八、考核：明确“怎么评分”，知道努力方向
考核分4部分，总分100分，新手要重点抓“每周报告”和“答辩”：

| 考核项目 | 分值 | 要求 | 新手建议 |
|----------|------|------|----------|
| 每周报告 | 30分（5分×6周） | 每周写一次“进度报告”（比如这周做了商品列表，遇到了什么问题） | 每周花1小时写，别攒到最后，老师看的是“过程” |
| 答辩 | 60分（核心） | 1. 演示程序：打开你的系统，操作一遍（比如看商品、加购物车、结算）；<br>2. 展示代码：打开代码，讲关键部分（比如折扣计算的代码）；<br>3. PPT讲解：讲你的设计思路（比如用了C/S框架，数据库建了哪些表）；<br>4. 回答问题：老师问技术细节（比如“怎么解决并发问题”） | 提前练3遍演示流程，别紧张；代码里标上注释，讲的时候能找到地方 |
| 大报告 | 10分 | 写一份完整的“课程设计报告”（包括需求、设计、实现、总结） | 答辩后根据演示内容补写，不难，主要是“记录全过程” |
| 额外加分/扣分 | - | 加分：有3个以上难点（比如自己实现了复杂折扣、解决了并发问题）、个人进步大；<br>扣分：功能极简（只做了最基础的，没扩展）、没满足全部要求 | 尽量多做一个小功能（比如退货），算“难点”，容易加分 |


# 新手总结：先做什么，再做什么
1. 第一步：确定技术栈（比如“客户端用Java，服务器用Java，数据库用MySQL”）。
2. 第二步：画简单的设计图（比如“用户买商品”的流程，数据库要建哪几张表）。
3. 第三步：分模块实现（先做“显示商品列表”，再做“购物车”，最后做“折扣和结算”）。
4. 第四步：每周写报告（记录进度和问题，不懂就问老师）。
5. 第五步：答辩前练演示（确保程序能跑通，代码能讲清楚）。

按这个步骤来，新手也能完成课程设计，重点是“别贪多，先实现核心功能，再慢慢优化”。
